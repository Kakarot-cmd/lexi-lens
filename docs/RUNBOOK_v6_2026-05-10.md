# Mistral Swap + Cache v6 — Drop-in Runbook
**Lexi-Lens v6.0 · 2026-05-10**

This is the operator's guide for shipping the Mistral-primary swap and the
cache v6 redesign as one coordinated release. Six code files plus one
migration. No client-side changes; this is entirely Edge Function + DB.

---

## What changes for kids

- Cache hits get **real kid voice** instead of reconstructed-from-reasoning
  feedback. The `kid_msg.young` / `kid_msg.older` strings are written by
  the model on the first scan of a property and replayed on every cache
  hit thereafter, age-banded to the child.
- Quest atmosphere shows up in feedback (e.g. *"❄ The Ice Dragon shimmers
  as you find another truth."*) when `quests.feedback_flavor_template` is
  populated. Compose-only — zero cache cost.
- Faster scans on cache miss because Mistral median latency is ~38% lower
  than Haiku.

## What changes for the operator

- Single `feature_flags` row flips the entire eval primary. Every flip
  writes an `admin_audit_log` row automatically.
- `tier_config.haiku_calls_per_day` is now `primary_calls_per_day`. Existing
  values preserved; semantically the cap now gates "primary calls before
  fallback to Gemini".
- Cache keys are readable: `staging:lexi:v6:verdict:apple:round` instead
  of a btoa hash. `redis-cli SCAN MATCH 'staging:lexi:v6:verdict:apple:*'`
  works.

## What does NOT change

- Other Claude-using Edge Functions (`generate-quest`, `classify-words`,
  `retire-word`, `export-word-tome`) stay on Haiku 4.5. Out of scope for
  v6.0; can be migrated independently later.
- `EvaluationResult` public shape (the response body kids' devices receive)
  is unchanged. `properties` is still `PropertyScore[]` with verdict-only
  fields; `kid_msg`/`nudge` are internal and stripped before the response
  goes out.
- The `report-verdict` Edge Function. No change.
- `scan_attempts` schema. No new columns.

## Required env vars (staging + prod)

`MISTRAL_API_KEY` is new. The other two should already be set:

```cmd
:: From the project root, in Windows CMD:
supabase secrets set --project-ref <staging-ref> MISTRAL_API_KEY=mist-xxx
supabase secrets set --project-ref <prod-ref>    MISTRAL_API_KEY=mist-xxx
```

Optional:

- `MISTRAL_MODEL_ID` — defaults to `mistral-small-2603` (Mistral Small 4).
  Override only if you want to pin a specific snapshot.
- `MISTRAL_REASONING_EFFORT` — defaults to `none`. Bump to `low` if you
  see verdict quality regress during staging bake (unlikely; the eval
  already validated `none`).

Existing keys (no change):
- `ANTHROPIC_API_KEY` — required as deeper fallback.
- `GOOGLE_AI_STUDIO_KEY` — required as Gemini fallback.
- `UPSTASH_REDIS_REST_URL`, `UPSTASH_REDIS_REST_TOKEN`.
- `CACHE_ENV_NAMESPACE` — should be set per project (`staging` / `prod`).

If you don't set `MISTRAL_API_KEY`, the Edge Function falls back to Gemini
on every primary-budget pick, then to Haiku if Gemini is also missing.
The fallback is logged but the system keeps working — useful during the
deploy window if there's a brief gap before the secret lands.

## Deploy order

The migration and the Edge Function deploy MUST happen in the same
operation. Open two windows.

### Staging first

**Window A — Supabase Dashboard SQL Editor (staging project)**, paste the
full `20260510_mistral_swap_and_cache_v6.sql` contents and have it ready
to run.

**Window B — Windows CMD**, ready to run:

```cmd
supabase functions deploy evaluate --project-ref <staging-ref>
```

Then:

1. **Set the Mistral key first.** This is the only ordering that matters:
   ```cmd
   supabase secrets set --project-ref <staging-ref> MISTRAL_API_KEY=mist-xxx
   ```

2. **Apply the migration.** Click Run in Window A. The migration:
   - Renames `tier_config.haiku_calls_per_day` → `primary_calls_per_day`
   - Adds `quests.feedback_flavor_template`
   - Installs the `feature_flags_audit_trg` trigger
   - Flips `evaluate_model_provider` to `'mistral'` (this is the live cutover)

3. **Deploy the Edge Function.** Within ~30 seconds of step 2, run Window B.
   Until this lands, the previous deploy of evaluate is still running
   against the renamed column — it'll fall through to the hardcoded floor
   for tier limits, which is safe but noisy.

4. **FLUSH the v5 cache namespace.** With Mistral now primary, the v5
   entries (Haiku-produced, no kid_msg) are noise. Two ways:

   **Soft option — let v5 entries expire naturally.** The 14-day TTL means
   they drain out automatically. New writes go to the v6 namespace
   (`staging:lexi:v6:*`), so the v5 keys (`staging:lexi:eval:*`) just sit
   there unused.

   **Hard option — purge immediately.** From the Upstash console (or
   redis-cli), in the staging Redis instance:
   ```
   SCAN 0 MATCH staging:lexi:eval:* COUNT 1000
   :: For each key returned, DEL it. Or use UNLINK in batches.
   ```
   Cleaner, but with a tiny cache the soft option is fine.

5. **Verify.** Run a fresh scan from the beta build pointed at staging.
   Edge Function logs should show:
   ```
   [evaluate] cache: cached=0 missing=2 (FULL MISS) childId=...
   [evaluate] routing: model=mistral-small-4 reason=primary-budget parent_tier=free primary_today=0/3 childId=...
   ```
   Then a second scan of the same object:
   ```
   [evaluate] cache: cached=2 missing=0 (FULL HIT) childId=...
   [evaluate] resolved-name cache: HIT (producedBy=mistral-small-4, name="...") childId=...
   ```
   The cache hit produces real kid voice now — verify `childFeedback` in
   the response is shaped like Mistral's tone, not the templated
   "You found something X!" form.

6. **Bake on staging.** Suggested ≥ 24 hours of real scans. Watch for
   `CACHE_SHAPE_INVALID` log lines — they should never appear.

### Production

Same five steps. Only flip prod after staging has been clean for ≥ 24
hours.

## Verification queries

After the migration applies, run these in Supabase Dashboard SQL Editor:

```sql
-- 1. Confirm column rename
SELECT column_name FROM information_schema.columns
WHERE table_schema = 'public' AND table_name = 'tier_config'
  AND column_name IN ('haiku_calls_per_day', 'primary_calls_per_day');
-- Expect: only primary_calls_per_day

-- 2. Confirm constraint rename
SELECT conname FROM pg_constraint
WHERE conrelid = 'public.tier_config'::regclass;
-- Expect: tier_config_primary_le_cap (NOT tier_config_haiku_le_cap)

-- 3. Confirm flag flip
SELECT key, value, updated_at FROM public.feature_flags
WHERE key = 'evaluate_model_provider';
-- Expect: value='mistral'

-- 4. Confirm audit row
SELECT created_at, action, payload FROM public.admin_audit_log
WHERE action = 'feature_flag_change'
ORDER BY created_at DESC LIMIT 1;
-- Expect: payload.old_value previous, payload.new_value='mistral'

-- 5. Confirm new quests column
SELECT column_name FROM information_schema.columns
WHERE table_schema = 'public' AND table_name = 'quests'
  AND column_name = 'feedback_flavor_template';
-- Expect: one row

-- 6. Spot-check cache (Upstash console SCAN, not SQL)
:: SCAN 0 MATCH staging:lexi:v6:verdict:* COUNT 100
:: Expect: keys like "staging:lexi:v6:verdict:apple:round"
```

After ~10 production scans, run this to confirm `model_id` is being
populated correctly:

```sql
SELECT model_id, count(*) AS n
FROM public.scan_attempts
WHERE created_at >= now() - interval '1 hour'
  AND cache_hit = false
GROUP BY 1
ORDER BY n DESC;
-- Expect: model_id='mistral-small-4' dominant
```

## Eval harness re-run (post-swap validation)

You confirmed earlier you'd re-run the existing harness. Mistral is now
the production primary, so a small change to the harness compares
production-against-eval rather than picking models. Indicative command —
adjust to your harness's CLI:

```cmd
deno run -A scripts/eval-adapters.ts --providers mistral --corpus free_dungeon
```

The 40 free_dungeon entries that produced the 94.5% post-audit-v3
agreement number should now produce ~100% self-agreement when run through
production prompt construction (which v6 evaluateObject is). Anything
< 100% means the prompt rewrite for v6 changed Mistral's verdicts on the
audit corpus; investigate.

## Rollback

If you need to roll back the model:

```sql
UPDATE public.feature_flags
SET    value      = 'anthropic',
       updated_at = now()
WHERE  key        = 'evaluate_model_provider';
```

~60 seconds for warm Edge Function containers to pick up the new value.
After that, the kill-switch path in `tierRouting.ts` routes everyone to
Haiku regardless of tier or budget.

Cached entries (Mistral-produced) keep serving via cache hit until 14-day
TTL drift. Acceptable because Mistral and Haiku verdicts agree 94.5%+.
If you want a clean cache state on rollback, FLUSH the v6 namespace and
let it organically rewarm on Haiku.

If you need to roll back the schema (column rename, audit trigger, quests
column), the inverse SQL is in the migration file's footer comment.

### Rollback triggers (decide in advance)

Roll back automatically if any of:

- `verdict-report` rate spikes >3× baseline for >2 hours
- Mistral p95 latency >5s sustained (eval baseline 2.34s)
- Adapter call-failure rate >5% over 30 minutes
- `CACHE_SHAPE_INVALID` log lines >10/minute (writer is shipping malformed
  entries; investigate but the system is still safe — invalid entries are
  treated as miss)

## What's deferred

These came up during planning and are explicitly out of scope for this PR:

- **`scripts/prewarm-cache.ts`** updates. The script needs `mistral` added
  to its CLI provider whitelist and the new v6 value shape on writes. We
  decided to skip prewarm entirely until post-launch organic scan data
  shows what's worth warming. Update the script when you actually want
  to run it.

- **`test/evaluateHandler.ts`** stale `buildCacheKey` 3-arg suite. The
  production `index.ts` no longer calls a `buildCacheKey` function with
  questId/pendingWords. The dead test should be replaced with a
  `buildPerPropCacheKey` + `cacheGetVerdict` shape-validation suite that
  matches v6 production. Independent debt; safe to defer.

- **`get_evaluate_context` RPC field rename.** The RPC still returns a
  field literally named `haiku_calls_today`. The Edge Function reads it
  and immediately renames to `primaryCallsToday`. Cosmetic; the count
  itself is correct (cache_hit=false scans). Optional cleanup.

- **`docs/MODEL_PROVIDER_SWAP_2026-05-08.md`** is still in the repo. It
  documents the previous swap (Gemini variant flip) and is now stale.
  Either delete or annotate as superseded; this v6.0 doc is the active
  runbook.

- **Sentry breadcrumb wiring on Edge Function side.** The
  `CACHE_SHAPE_INVALID` log line is a stand-in. Edge Functions don't have
  the Sentry SDK directly — adding raw-fetch instrumentation to Sentry's
  HTTP API is a 50-line diff for a separate PR.

- **Quest flavor template seeding.** Migration adds the column but doesn't
  populate it. Decide per-quest whether the atmosphere is worth writing.
  A handful of templates is fine — the feature is "lit up" the moment any
  quest has a non-NULL value.
