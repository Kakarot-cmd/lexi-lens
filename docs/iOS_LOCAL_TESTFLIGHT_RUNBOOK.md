## The recipe (canonical, after fresh prebuild)

This is the sequence that produces a TestFlight-ready archive when the `ios/` folder has been freshly regenerated by `expo prebuild`. Some steps are skippable on incremental builds — flagged inline.

### Phase 1 — Prebuild and pod setup (~3 min)

```bash
cd ~/projects/lexi-lens

# Regenerate ios/ from scratch. Required when:
# - Switching APP_VARIANT (e.g. dev <-> staging)
# - Native dependency changes
# - Recovering from a corrupted ios/ state
APP_VARIANT=staging npx expo prebuild --platform ios --clean
```

**At this point, four manual edits MUST be applied to the regenerated `ios/` before pod install will produce a buildable project.** Until config plugins exist for each, these are hand-edits every time.

### Phase 2 — Apply the four Xcode 26 fixes (~5 min)

> **AUTOMATED since 2026-05-30.** `plugins/withXcode26Compat.js` (registered in
> `app.config.js`) re-applies Fixes #1, #2, and #3 automatically during
> `expo prebuild`. After a prebuild you should see three `[withXcode26Compat]`
> log lines. **Verify with Phase 3, then skip straight to `pod install`.** The
> manual steps below are retained as the reference spec the plugin implements,
> and as the fallback if the plugin ever fails to apply (e.g. an Expo template
> change moves the Podfile anchor — the plugin logs a clear WARNING in that
> case rather than failing silently). Fix #4 (open the workspace, not the
> xcodeproj) is a human action in Xcode and is NOT automatable — still on you.

#### Fix 1 — Podfile post_install (Xcode 26 + RN 0.81 compat)

Edit `ios/Podfile`. Find `post_install do |installer|` block. Replace from that line through its matching closing `end` with this block (keep everything else untouched, especially the bottom-most `end` that closes the target):

```ruby
  post_install do |installer|
    react_native_post_install(
      installer,
      config[:reactNativePath],
      :mac_catalyst_enabled => false,
      :ccache_enabled => ccache_enabled?(podfile_properties),
    )

    # Xcode 26 + React Native 0.81 archive compatibility
    # Ref: https://reactnative.dev/blog/2025/08/12/react-native-0.81
    # Ref: https://github.com/facebook/react-native/releases/tag/v0.81.1
    installer.pods_project.targets.each do |target|
      target.build_configurations.each do |config|
        # THE Xcode 26 archive fix — disables Swift explicit modules on all
        # Pod targets. Without this, PrecompileSwiftBridgingHeader fails with
        # "modulemap not found" errors for ~25 Pod targets.
        config.build_settings['SWIFT_ENABLE_EXPLICIT_MODULES'] = 'NO'

        # Force c++17 on fmt + RCT-Folly. Without this, fmt::format_to fails
        # under c++20 with consteval errors in Xcode 26.
        # Ref: facebook/react-native#55601
        if ['fmt', 'RCT-Folly'].include?(target.name)
          config.build_settings['CLANG_CXX_LANGUAGE_STANDARD'] = 'c++17'
        end
      end
    end
  end
```

#### Fix 2 — User script sandboxing

```bash
sed -i '' 's/ENABLE_USER_SCRIPT_SANDBOXING = YES/ENABLE_USER_SCRIPT_SANDBOXING = NO/g' \
  ios/LexiLensStaging.xcodeproj/project.pbxproj
```

(Substitute `LexiLensDev` if you prebuilt with `APP_VARIANT=development`.)
Without this, the "Bundle React Native code and images" build phase fails because it can't write `ip.txt`.

#### Fix 3 — `.xcode.env.local`: node path + Sentry symbol upload bypass

```bash
echo "export NODE_BINARY=\"$(command -v node)\"" >> ios/.xcode.env.local
echo "export SENTRY_DISABLE_AUTO_UPLOAD=true" >> ios/.xcode.env.local
```

**NODE_BINARY (Sharp Edge #5):** Without an absolute `NODE_BINARY` here, an
archive launched from Finder/Dock (no nvm node inherited) dies in the Hermes
"Replace Hermes" build phase with `line 9: : command not found`. See Sharp
Edge #5 for the full mechanism. The plugin writes this automatically using
`process.execPath`; the manual line above is only for a no-plugin fallback —
and `$(command -v node)` only resolves correctly when run from a terminal
where node is on PATH, so prefer the plugin.

**SENTRY_DISABLE_AUTO_UPLOAD:** Without this, archive succeeds compile but
fails at the final "Upload Debug Symbols to Sentry" build phase because no
`SENTRY_AUTH_TOKEN` is configured. Non-fatal long-term (we want to upload
symbols eventually) but blocks the build.

#### Fix 4 — Install pods

```bash
cd ios && pod install && cd ..
```

Wait for "Pod installation complete!" line — should report 111 dependencies, 132 total pods.

### Phase 3 — Verify the fixes landed (~30 sec)

```bash

echo "SWIFT_ENABLE_EXPLICIT_MODULES NO count:"
grep -c "SWIFT_ENABLE_EXPLICIT_MODULES = NO" ios/Pods/Pods.xcodeproj/project.pbxproj

echo "ENABLE_USER_SCRIPT_SANDBOXING NO count:"
grep -c "ENABLE_USER_SCRIPT_SANDBOXING = NO" ios/LexiLensStaging.xcodeproj/project.pbxproj

echo "NODE_BINARY set (absolute path):"
grep "NODE_BINARY" ios/.xcode.env.local

echo "SENTRY_DISABLE_AUTO_UPLOAD set:"
grep -c "SENTRY_DISABLE_AUTO_UPLOAD" ios/.xcode.env.local
```

Expected output: ~290 / ~3 / 1 (or higher for SWIFT_ENABLE...).

### Phase 4 — Open Xcode and archive (~20 min)

```bash
# Force-close any cached Xcode instance to avoid workspace-vs-xcodeproj confusion
killall Xcode 2>/dev/null
sleep 3

# Open the WORKSPACE (not the xcodeproj). This matters — see Sharp Edge #4.
open ios/LexiLensStaging.xcworkspace
```

In Xcode:

1. **Verify left sidebar shows BOTH `LexiLensStaging` AND `Pods` projects.** If you only see one, you've opened the wrong file.
2. **Top bar:** scheme = `LexiLensStaging`, destination = "Any iOS Device (arm64)" (NOT a simulator).
3. **Edit Scheme → Run → Info tab:** Build Configuration = `Release`.
4. **Build number (CFBundleVersion):** set it in **`app.config.js` → `ios.buildNumber`** BEFORE prebuild, not in Xcode. The Xcode General-tab "Build" field is wiped by the next `prebuild --clean` (ios/ is gitignored CNG), which is why a fresh-prebuild archive silently reverts to `(1)` and gets rejected by ASC as a duplicate. The value must be unique and higher than any build already uploaded for the current `version` string. (If you must fix an already-prebuilt archive without re-prebuilding, see the in-place override below.)
5. **Signing & Capabilities tab:** "Automatically manage signing" checked, Team = Naveen Juttiga (9A8R9ZM546).
6. **Product → Clean Build Folder** (Shift+Cmd+K). Optional but recommended on first build.
7. **Product → Archive.** Wait 15-25 min for a clean build, 1-2 min for incremental.

#### Build-number quick-fix (already prebuilt, don't want to re-prebuild)

If `ios/` is already generated and you only need to correct the build number
on an archive (e.g. it came out `(1)` and ASC will reject it as a duplicate),
override CFBundleVersion in place and re-archive — no prebuild, no pod install:


Then Product → Archive again. This is a one-shot patch; still set
`ios.buildNumber` in `app.config.js` so the value survives the next prebuild.

## The four Xcode 26 sharp edges (root cause reference)

These are the things that made the first archive take 9 hours. Documenting so future builds don't re-discover them.

### Sharp Edge #1 — fmt consteval under c++20

**Symptom:** Archive (or `expo run:ios`) fails with `call to consteval function 'fmt::basic_format_string<...>' is not a constant expression` errors in fmt headers.

**Cause:** Xcode 26 defaults to c++20. fmt's consteval-based format string validation breaks under c++20 with Apple's clang. Reported as facebook/react-native#55601.

**Fix:** Pin fmt and RCT-Folly to c++17 (covered in Fix #1).

### Sharp Edge #2 — Xcode 26 Explicit Modules + CocoaPods incompatibility

**Symptom:** Archive fails very fast (~3 seconds) with hundreds of `module map file '...XXX.modulemap' not found` errors. All in the `ArchiveIntermediates/.../BuildProductsPath/Release-iphoneos/<PodName>/` directory.

**Cause:** Xcode 26 enabled `SWIFT_ENABLE_EXPLICIT_MODULES = YES` by default. This conflicts with how CocoaPods generates module maps. Per Apple's React Native team in the [0.81 release blog](https://reactnative.dev/blog/2025/08/12/react-native-0.81): the workaround is to set this flag to NO. React Native 0.81.1 baked this in for projects using precompiled XCFrameworks — but **this project uses `buildReactNativeFromSource: true`** (Expo workaround for an unrelated SDK 54 TestFlight issue), which bypasses the 0.81.1 auto-fix.

**Trap during debugging:** Xcode's UI has a setting labeled "Explicitly Built Modules" wired to `SWIFT_USE_INTEGRATED_DRIVER`. This is NOT the same setting. Toggling it accomplishes nothing for this issue. The actual setting (`SWIFT_ENABLE_EXPLICIT_MODULES`) doesn't appear in Xcode's Build Settings UI as a discoverable row — has to be applied via Podfile.

**Fix:** Apply `SWIFT_ENABLE_EXPLICIT_MODULES = NO` to all Pod targets via Podfile post_install (covered in Fix #1).

### Sharp Edge #3 — Sentry symbol upload requires auth, archive fails without it

**Symptom:** Archive completes ~10,000 lines of successful compilation, all 165 targets build, Hermes bundle generates, sourcemaps generate — then dies at the final script phase with `error: Auth token is required for this request. Please run sentry-cli login and try again!`

**Cause:** The `@sentry/cli` package added a "Upload Debug Symbols" build phase. Without `SENTRY_AUTH_TOKEN` (or `SENTRY_DISABLE_AUTO_UPLOAD=true`) in the build environment, the upload fails. Command-line env vars passed via `expo run:ios` don't propagate to Xcode Archive (different process tree).

**Fix:** Add `export SENTRY_DISABLE_AUTO_UPLOAD=true` to `ios/.xcode.env.local` (covered in Fix #3).

**Long-term:** Configure `SENTRY_AUTH_TOKEN` properly so symbols actually upload — needed for usable Sentry crash reports in production. Not blocking for TestFlight.

### Sharp Edge #4 — Opening xcodeproj instead of xcworkspace breaks Pods integration

**Symptom:** Archive fails with `Target dependency graph (1 target)` showing only `LexiLensStaging` with `(no dependencies)` — Pods are completely absent from the build graph. Same modulemap errors as Sharp Edge #2 but for a totally different reason.

**Cause:** Somewhere in the session, Xcode lost workspace context and we were inadvertently editing/archiving `LexiLensStaging.xcodeproj` directly. CocoaPods integration only works when the **workspace** (`.xcworkspace`) is open, because the workspace is what bundles both `LexiLensStaging.xcodeproj` AND `Pods/Pods.xcodeproj` together.

**Visual diagnostic:** Open the project navigator (cmd+1 in Xcode). You should see TWO blue project icons at the top level: `LexiLensStaging` AND `Pods`. If you only see one, you have the xcodeproj open, not the workspace.

**Definitive log diagnostic:** Search the build log for "Target dependency graph". A healthy build shows `(165 targets)` with many implicit dependencies listed. A broken one shows `(1 target)`.

**Fix:** Force-close Xcode, reopen via `open ios/LexiLensStaging.xcworkspace`.

### Sharp Edge #4b — pod deintegrate as a nuclear option

If the build log shows "1 target" even with the workspace properly open, the CocoaPods integration in the xcodeproj is silently broken (cached build state, stale references, etc.). Recovery:

```bash
cd ios
pod deintegrate    # removes all CocoaPods integration from .xcodeproj
pod install        # recreates it from scratch
cd ..
```

This was the actual fix on May 24 — even after correcting the workspace-vs-xcodeproj confusion, the dependency graph stayed broken until deintegrate.

### Sharp Edge #5 — Hermes script dies with `line 9: : command not found` (NODE_BINARY empty)

**Symptom:** Archive fails almost immediately in the `hermes-engine` target.
Build log shows the `[Hermes] Replace Hermes for the right configuration`
script phase (`.../hermes-engine.build/Script-XXXX.sh`) failing:

```
[Warning] You need to configure your node path in the ".xcode.env" file ...
Now using node v20.20.2 (npm v10.8.2)
.../hermes-engine.build/Script-XXXX.sh: line 9: : command not found
Command PhaseScriptExecution failed with a nonzero exit code
```

**Cause:** React Native's `scripts/xcode/with-environment.sh` initialises
`NODE_BINARY=$(command -v node || echo "")`. When Xcode is launched from
Finder/Dock (rather than a terminal that has nvm's node active), `command -v
node` returns **empty**. The auto-generated `ios/.xcode.env` uses the same
empty `$(command -v node)`, and `ios/.xcode.env.local` (before this fix) only
held `SENTRY_DISABLE_AUTO_UPLOAD`. So `NODE_BINARY` stays empty, and
`with-environment.sh` falls back to `scripts/find-node-for-xcode.sh`, which
sources nvm and puts `node` on PATH — that is the misleading "Now using node
v20.20.2" line — but **never sets `NODE_BINARY`**. The Hermes script then runs
`"$NODE_BINARY" .../replace_hermes_version.js` with `NODE_BINARY=""`, so bash
tries to execute an empty command word → `line 9: : command not found`.

**Why it's intermittent:** if Xcode was launched from a Terminal where nvm's
node was already on PATH (or you ran `expo run:ios` from that shell first),
`command -v node` resolves and the archive succeeds. The first 1.0.31 archive
likely got lucky this way, which is why this edge wasn't caught until a clean
fresh-prebuild archive from Finder.

**Fix:** Pin `NODE_BINARY` to an **absolute** path in `ios/.xcode.env.local`,
which `with-environment.sh` sources after `.xcode.env`, so it wins and the
fragile fallback is never entered. `plugins/withXcode26Compat.js` now writes
this automatically (Fix #3a) using `process.execPath` — the node running
prebuild on your Mac — and rewrites it on every prebuild so it self-heals
across node-version upgrades. Manual fallback (no plugin), run from a terminal
with node on PATH:

```bash
echo "export NODE_BINARY=\"$(command -v node)\"" >> ios/.xcode.env.local
```

**Confirmation the fix is working:** the archive log line changes from the
warning + "Now using node" to `Node found at: /Users/<you>/.../bin/node`, and
`grep NODE_BINARY ios/.xcode.env.local` shows an absolute path.

## Build configuration / variant matrix

What runs where, what env vars load where. This is a frequent source of confusion.

| APP_VARIANT | Bundle ID | xcworkspace name | RevenueCat keys | Supabase | When to use |
|---|---|---|---|---|---|
| `development` | `com.navinj.lexilens.dev` | `LexiLensDev.xcworkspace` | `_STAGING` keys | Staging | `expo run:ios` to local device for iteration |
| `staging` | `com.navinj.lexilens` | `LexiLensStaging.xcworkspace` | `_STAGING` keys | Staging | TestFlight Internal Testing builds |
| `production` | `com.navinj.lexilens` (same!) | `LexiLens.xcworkspace` | `_PROD` keys | Production | Public App Store releases (not yet attempted via local archive) |

**Key insight:** `APP_VARIANT` only affects the bundle ID and project name. It does NOT pick the RevenueCat / Supabase keys — those are picked at runtime by `env.ts` based on `EXPO_PUBLIC_APP_VARIANT`.

**Build Configuration (Debug vs Release) is orthogonal** to APP_VARIANT. Both Debug and Release builds of a `staging` variant use staging RevenueCat keys. Release just enables optimizations, JS minification, dev assertion stripping.

**For the 1.0.31 TestFlight build:**
- APP_VARIANT = `staging` → bundle `com.navinj.lexilens`, staging keys
- Build Configuration = `Release` → optimized, production-grade build artifact
- RevenueCat: **enabled, pointed at staging RevenueCat project** (the one with `default_prod` offering, 4 products, premium entitlement)
- Supabase: **staging** (`zhnaxafmacygbhpvtwvf`)

This is exactly the build needed to execute Phase 4.4l (sandbox IAP testing).

## What's fragile (TODOs for Monday or later)

The current state requires four manual edits after every `expo prebuild --clean`. This is fragile and forgettable. Each should become a durable config plugin:

### TODO 1 — `plugins/withXcode26Compat.js` ✅ DONE 2026-05-30

Built and registered in `app.config.js`. Single Expo config plugin applying on every prebuild:
- Fix #1 via `withPodfile` — injects the per-Pod-target loop INSIDE the existing `post_install` block (after `react_native_post_install(...)`), setting `SWIFT_ENABLE_EXPLICIT_MODULES = NO` on all Pod targets and `CLANG_CXX_LANGUAGE_STANDARD = c++17` on fmt + RCT-Folly.
- Fix #2 via `withXcodeProject` — sets `ENABLE_USER_SCRIPT_SANDBOXING = NO` on every build configuration of the **app** project (variant-agnostic; no hard-coded `LexiLensStaging` name, so it works for dev/staging/production alike). Structural edit, not a sed.
- Fix #3 via `withDangerousMod('ios', …)` — writes `ios/.xcode.env.local`: an absolute `NODE_BINARY` (from `process.execPath`, rewritten each prebuild — Sharp Edge #5) plus `SENTRY_DISABLE_AUTO_UPLOAD=true` (appended if absent).

If the Podfile anchor ever moves, Fix #1 logs a WARNING and passes through — the archive then fails with a clear modulemap error, the documented fallback being the manual Phase 2 Fix #1 edit. Fix #4 (workspace not xcodeproj) is intentionally NOT automated — it is a human action in Xcode.

### TODO 2 — Hermes dSYM upload fix ✅ DONE 2026-05-30

The "Upload Symbols Failed" warning on every archive. Automated by
`withXcode26Compat.js` Fix #4 — a "Copy Hermes dSYM into archive" run-script
build phase on the app target.

**Path correction (verified against RN 0.81.5):** the prebuilt Hermes ships
iOS as an **XCFramework**, so the dSYM is *inside* it
(`Pods/hermes-engine/destroot/Library/Frameworks/universal/hermes.xcframework/ios-arm64/dSYMs/hermes.framework.dSYM`),
NOT at the `universal/hermes.framework.dSYM` path noted earlier — that path
doesn't exist. The plugin locates it with `find` (excluding the simulator
slice, whose UUID won't match a device archive) and copies it into
`${DWARF_DSYM_FOLDER_PATH}` during archive. Guarded so it no-ops on
non-archive builds.

Alternatively, when SENTRY_AUTH_TOKEN is configured, do the upload via `@sentry/react-native` which handles Hermes dSYM correctly.
